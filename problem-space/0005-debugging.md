## Summary
[summary]: #summary

The quality of debug information in mixed Rust/C++ projects is low, leading to difficulty debugging some code.

Rust uses C++-style debug info to interact with some debuggers, but this has drawbacks where the language semantics are different.
Less debugging is required for pure Rust projects, but when mixing Rust with C++, it is used a lot more.

## Impact
[impact]: #impact

The following impacts have been reported by users:

- It is difficult to run the Rust `Display` impl from `lldb` (from a user of `cxx` for FFI)
- There are highlighting discrepancies
- There is no first-class support for common types, e.g. break on an `Option<T>` doesn't work
- When in Rust debug mode, the representation of a C++ type is less helpful
- Rust provides low-quality `.pdb` output on Windows
- Debugging support for Rust async varies based on the executor
- Post-mortem debugging needs stack traces to show up, regardless of language
  - Rust standard library does not fall back to the symbol table if DWARF debug info is absent

## Acceptance Criteria
[acceptance-criteria]: #acceptance-criteria

- Debuggers (e.g., LLDB) can display common Rust types such as `Option<T>` and `Result<T, E>` in a human-readable, language-aware format rather than raw memory representations.

- Rust trait-based formatting (e.g., `Display` or `Debug`) can be invoked or emulated within the debugger to present values meaningfully.

- Breakpoints can be set on high-level Rust constructs, including enum variants and pattern-matched conditions.

- Mixed Rust and C++ stack traces are consistently available and readable, with clear symbol resolution across language boundaries.

- Debug information generated by Rust (e.g., DWARF on Unix, `.pdb` on Windows) is sufficiently complete to support accurate inspection of variables and types.

- Debugging asynchronous Rust code produces stable and understandable stack traces, independent of the executor used.

## Further Background
[further-background]: #further-background

Modern debuggers rely on debug information formats such as DWARF (on Unix-like systems) and PDB (on Windows) to map compiled binaries back to source-level constructs.

Rust currently emits debug information that is largely compatible with existing tooling designed for C and C++. However, Rust’s language features—such as enums with rich semantics, pattern matching, and async/await transformations—do not always map cleanly onto these formats.

As a result, debuggers often fall back to displaying low-level representations of Rust types, exposing implementation details (e.g., discriminants and internal fields) rather than high-level abstractions.

In mixed-language projects, these issues are amplified. C++ and Rust use different abstractions and conventions, and there is no unified model for representing types and execution flow across both languages in debugging tools.

Additionally, Rust’s async model introduces compiler-generated state machines, which can obscure control flow and make stack traces harder to interpret during debugging.

## Unresolved questions
[unresolved-questions]: #unresolved-questions

- What is the best way to represent Rust-specific abstractions (e.g., enums, traits, async state machines) in a way that is both accurate and compatible with existing debugger infrastructure?

- Should improvements be made primarily in Rust’s debug information generation, or in debugger tooling (e.g., LLDB and GDB), or both?

- How can a consistent debugging experience be provided across platforms, particularly given differences between DWARF and PDB formats?

- What level of support is feasible for invoking Rust-specific functionality (e.g., trait methods like `Display`) within a debugger?

- How can async debugging be standardized across different executors and runtimes?

- What fallback mechanisms should be used when full debug information is unavailable (e.g., relying on symbol tables)?
